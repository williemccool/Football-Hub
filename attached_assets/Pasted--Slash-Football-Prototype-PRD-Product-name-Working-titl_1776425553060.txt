# Slash Football Prototype PRD

## Product name

Working title: **Slash Football**

## Product vision

Build a mobile-first football club simulation game where players repeatedly return for ultra-short **slash scouting runs**, collect player shards and traits, build a club, let that club compete in asynchronous simulated ([docs.replit.com](https://docs.replit.com/replitai/web-apps?utm_source=chatgpt.com)) matches.

This is **not** a traditional football manager clone.
This is a **collection-driven football club RPG/simulator** with a highly repeatable casual loop.

## Prototype objective

The first prototype must answer one question:

**Will players enjoy the loop of scouting via slash gameplay, upgrading a club, and watching the consequences in simulated football matches?**

The first prototype is successful if we can validate:

1. The slash loop is fun enough to repeat.
2. Squad progression feels meaningful.
3. Simulated match outcomes feel understandable.
4. Highlight reels feel rewarding enough to bring players back.
5. The entire loop works well on mobile web inside a Replit-built product.

# 1. Platform decision

## Chosen platform

**Replit only** for the first prototype.

## What that means in practice

The first prototype can be built on Replit in **two valid formats**:

* a **native mobile app** using **React Native + Expo**
* a **responsive web companion/admin surface** for internal tools if needed

## Recommended prototype format

For this game, the recommended primary client is:

* **native mobile app first** for iOS and Android using React Native + Expo on Replit
* optional lightweight web surfaces for admin tooling, economy tuning, and internal testing

## Why this is now the preferred route

This game benefits from:

* better touch interaction for the slash mechanic
* stronger mobile feel for repeat 20-second sessions
* easier use of device-native patterns such as haptics, notifications, and full-screen interaction
* a cleaner path to eventual App Store and Play Store distribution

## What we are not building in prototype v1

* no Unity
* no full console-style realistic 3D football engine
* no real player or club licensing
* no multiplayer real-time manual gameplay
* no complicated transfer market

## Product format for prototype v1

* **frontend client:** native mobile app built on Replit with React Native + Expo
* **backend:** API + simulation engine hosted on Replit
* **database:** Replit SQL database
* **storage:** Replit App Storage for replay data, assets, and generated highlight payloads
* **deployment:** Replit deployment for backend, plus Expo-based native preview/distribution during prototype stage

# 2. Core game fantasy

The player fantasy is:

**Scout unknown talent, build a club identity, evolve your squad, and watch your football decisions turn into match moments.**

The emotional progression is:

1. I found this player.
2. I invested in this player.
3. I fixed my weak squad area.
4. I watched that improvement show up in the match.
5. I now want to return and improve more.

# 3. Core gameplay loop

## Primary loop

1. Player opens the game
2. Player uses available scouting tickets
3. Player plays a 20-second slash run
4. Player earns rewards
5. Player unlocks or upgrades players and traits
6. Player configures squad and tactics
7. Club participates in async fixtures
8. Player watches short highlight reel
9. Player claims rewards and season progress
10. Player returns later for more scouting runs

## Secondary loop

* complete daily objectives
* improve weak positions
* salvage duplicates
* unlock traits
* progress season track
* customize club visuals

## Long-term loop

* build a better club over time
* climb divisions or league ranks
* collect rarer archetypes
* evolve star players
* define club identity and play style

# 4. Target audience

## Primary audience

* mobile-first football fans aged roughly 14–30
* players who enjoy football fantasy but do not want deep spreadsheet management from the first minute
* players who enjoy short repeatable loops

## Secondary audience

* football manager fans who want a lighter habit game

# 6. Slash scouting mode design

## Purpose

This is the habit loop.
It must be fast, tactile, and rewarding.

## Session length

* 20 seconds

## Ticket system

* max tickets: 5
* refill cadence configurable in admin panel
* prototype default: 1 every 90 minutes

## Slash interaction

The player swipes through floating objects.

## Object types

### Positive objects

* player shard capsules
* midfielder shard orbs
* defender shard orbs
* attacker shard orbs
* trait fragments
* coin bundles
* training tokens
* evolution catalysts

### Negative objects

* red card traps
* fake agent files
* injury hazard markers
* burnout icons

### Special objects

* golden prospect pack
* role-targeted rare orb
* combo multiplier orb
* lucky salvage token

## Scoring rules

* each object gives point value and reward value
* combo chains increase reward multiplier
* hitting negative objects reduces multiplier or removes one reward slot
* higher score slightly improves end-of-run quality

## End-of-run output

After each run, present:

* total score
* rewards earned
* duplicate conversion result if applicable
* one CTA: upgrade player / play another run / go to squad

# 7. Economy design

## Currencies

### Soft currency

**Coins**
Used for:

* player upgrades
* trait unlock costs
* light crafting

### Progression currency

**Player Shards**
Used to unlock players and evolve them.

### Trait currency

**Trait Fragments**
Used to unlock or upgrade traits.

### Rare progression item

**Evolution Catalysts**
Used for higher-tier progression bands.

### Salvage currency

**Essence**
Obtained from duplicates.
Used for:

* targeted shard crafting
* rerolls
* evolution support

## Duplicate handling

When player gets a duplicate:

* extra shards add toward same player growth track if below cap
* if already capped, duplicate converts to:

  * coins
  * essence
  * optional small trait-fragment chance

## Prototype reward weighting

* generic shards: high frequency
* role-targeted shards: medium frequency
* trait fragments: medium-low frequency
* coins: medium frequency
* catalysts: low frequency
* full unlocks: very low frequency

## Design principle

Slash mode should **mostly generate fragments and progression materials**, not endless full player cards.

* deterministic
* explainable
* server-authoritative
* cheap to run

## Inputs

* team style
* pressing
* tempo

* chemistry
* fatigue
* randomness seed

## Outputs

* final score
* goalscorers
* cards

* chance quality count
* key event log
* player rating summary

## Event log structure

Each event should capture:

* minute
* event type
* team
* player ids involved
* pitch zone
* event importance
* camera hint
* commentary hint

## Event types

* chance created
* shot
* save
* goal
* yellow card
* red card
* interception
* counterattack
* set piece
* missed big chance

## Match cadence in prototype

* async fixtures generated on schedule
* user does not manually play match
* match resolves server-side when ready
* user later opens result and watches highlights

* duration: 20–45 seconds
* moments shown:

  * goals
  * saves
  * dangerous attacks
  * cards
  * late drama

## Visual style

* clean modern football broadcast UI
* mini pitch or stylized isometric field
* player circles or lightweight avatars
* ball movement animation
* score overlays
* celebration flash
* momentum bars

## Outcome screen

After highlights, show:

* full-time score
* key stat summary
* top performer
* squad weakness insight
* claim rewards button

# 19. Risks

## Risk 1

Slash gameplay is fun only once and becomes repetitive.

### Mitigation

* combo system
* targeted role rewards
* special event objects
* negative hazards
* visible progression purpose

## Risk 2

Sim outcomes feel random or unfair.

### Mitigation

* readable post-match explanations
* deterministic event generation
* squad weakness feedback

## Risk 3

Browser performance is weak on mobile.

### Mitigation

* stylized visuals
* 2D/2.5D rendering
* avoid heavy physics
* optimize animation loops

## Risk 4

Prototype becomes too big.

### Mitigation

* cut to one formation set, one season loop, one sim mode, one highlight style
