We already have a strong Slash Football build with:
- core gameplay loop
- season/league structure
- alpha-readiness work
- monetization foundation (shop, gems, season pass)
- analytics/event plumbing
- production-hardening groundwork

Do NOT rebuild the app.
Do NOT add random new gameplay systems.
Do NOT bloat the product.

Important constraints:
- keep the architecture portable and vendor-neutral
- do not hard-lock core logic to Replit-specific services
- keep monetization fair and cosmetics-first
- no pay-to-win
- preserve the identity of the product: slash -> collect -> upgrade -> squad -> match -> season

# GOAL

Take the current codebase and make it ready for:
- post-alpha iteration
- real beta / soft launch
- monetization activation readiness
- app-store submission preparation
- production v1 release candidate evaluation

This phase should focus on:
1. post-alpha balancing and usability improvements
2. monetization activation readiness
3. store-release plumbing
4. push/engagement readiness
5. release-candidate QA, docs, and operational safety

# PART 1 — POST-ALPHA PRODUCT AUDIT

Before making changes, audit the current product against what matters most for launch.

Review these areas:
- first-session completion
- onboarding drop-off
- slash run enjoyment and repeatability
- hazard frustration vs excitement
- squad upgrade clarity
- match fairness clarity
- highlight engagement
- league table excitement
- season progression clarity
- shop attractiveness
- pass attractiveness
- sync reliability
- error/crash risk
- economy pacing
- fairness of monetization

Create or update an internal audit doc that summarizes:
- launch strengths
- likely retention risks
- likely economy risks
- likely monetization risks
- likely store/submission risks
- recommended fixes applied in this pass

If live tester data exists in the codebase/docs, use it.
If real user data is not yet available, create a structured assumptions-based audit and make the app ready to ingest that data cleanly.

# PART 2 — FTUE / RETENTION / CLARITY IMPROVEMENTS

Refine the first-time and early-return experience.

## 2A. Tighten the first session
Improve the first session so users reach these moments quickly and clearly:
- first slash run
- first satisfying reward reveal
- first upgrade or unlock
- first lineup/tactics interaction
- first simulated match
- first highlight payoff
- first league-table consequence
- first clear “come back later” retention hook

Reduce confusion and dead time.

## 2B. Improve message hierarchy
Make the home/dashboard flow even clearer:
- strongest next best action
- why it matters
- what the user is working toward
- what reward is available now
- what they should come back for later

## 2C. Fairness and understanding
Improve how the app explains:
- why a hazard mattered
- why a match was won/lost
- why a player upgrade helped
- why the season table moved

Keep this concise, elegant, and rule-based. No bloated explanation walls.

# PART 4 — MONETIZATION ACTIVATION READINESS

The monetization structure exists. Now prepare it for real activation.

## 4A. Purchase architecture
Create a clean, portable purchase service abstraction.

Requirements:
- no hardcoding billing logic into UI
- product/catalog model remains separate
- support both placeholder mode and real store purchase mode
- prepare for Apple/Google in-app purchase adapters later
- support entitlement restoration conceptually
- safe handling of purchase success / failure / cancel / restore flows

Do not tightly couple business logic to a single vendor SDK.

## 4B. Entitlement model
Create a clear entitlement system for:
- owned cosmetics
- equipped cosmetics
- premium season pass access
- bundle ownership
- future founder pack / starter pack ownership

## 4C. Restore / purchase status UX
Add or improve:
- restore purchases placeholder/action
- purchase pending/failure states
- already-owned product messaging
- premium pass entitlement messaging
- account/sync reassurance around owned content

## 4D. Launch catalog readiness
Make the cosmetic and pass catalog feel launch-ready:
- better grouping
- rarity presentation
- collection progress where useful
- featured items / featured bundle presentation
- “coming soon” support for limited events
- clean product metadata structure

# PART 10 — WHAT NOT TO DO

Do not:
- add real-time multiplayer
- add new major gameplay systems
- add pay-to-win mechanics
- turn monetization predatory
- regress the core gameplay loop
- hard-lock the app to Replit-specific services or assumptions

# DEFINITION OF DONE

This phase is done when:
- the app is balanced and clearer after post-alpha review
- monetization is activation-ready structurally
- purchase/entitlement flows are architecturally sound
- re-engagement systems are launch-ready
- store/beta/soft-launch docs are in good shape
- release-candidate QA hardening is strong
- the product is genuinely close to a real soft launch / v1 release decision
